## 🪵 Troubleshooting Log Entries

### 🔴 [Issue #1] Firmware Uploads but Microcontroller is Unresponsive
* **Component / Subsystem:** System Startup / Memory Layout
* **Status:** ✅ Solved

#### 🔍 Symptom
* Firmware successfully compiles and flashes using `pyocd` without errors.
* After resetting or booting, the microcontroller remains completely dead (an LED connected to PA6 does not blink).
* No code in `main()` executes.

#### 🧠 Root Cause Analysis
Because the project was configured in PlatformIO with `board = disco_f030r8` (which represents an STM32F030R8), the compiler linked the default STM32F030R8 memory definitions:
1. **Linker Assumptions:** STM32F030R8 has **8KB of RAM** (RAM range: `0x20000000` to `0x20002000`).
2. **Initial Stack Pointer (`_estack`):** Set to the end of the 8KB RAM region: **`0x20002000`**.
3. **MM32G0001 Hardware Limit:** The MM32G0001 only has **2KB of RAM** (RAM range: `0x20000000` to `0x20000800`).
4. **Boot Failure Flow:** Upon power-on, the MM32G0001 loaded the stack pointer value `0x20002000`. On the first function call or stack write (such as inside the startup/assembly sequence), the CPU attempted to write to `0x20001FFC` (outside the physical 2KB limit). This triggered an immediate **Hard Fault / Bus Fault** and permanently froze the processor before reaching `main()`.

#### 🛠️ Resolution
1. **Created a Custom Linker Script (`custom.ld`):**
   * Configured the RAM length to **`2K`** and Flash length to **`16K`**.
   * This correctly forces `_estack` to be calculated at the valid RAM boundary of **`0x20000800`**.
   * Set `_Min_Heap_Size = 0x000` and `_Min_Stack_Size = 0x200` to conserve RAM.
2. **Configured PlatformIO:**
   * Appended `board_build.ldscript = custom.ld` in [platformio.ini](file:///C:/Users/AKSHAY/Desktop/code/projects/custom_servo/mm32_test/platformio.ini).
3. **Outcome:**
   * Binary compiled successfully, verified stack pointer alignment at `0x20000800`, and flashed onto the raw chip. The CPU now boots correctly and blinks the LED on **PA6**.

### 🟢 [Issue #2] USART1 Serial Output Shows NUL Bytes or Garbage on Serial Monitor
* **Component / Subsystem:** USART / Serial Communication
* **Status:** ✅ Solved

#### 🔍 Symptom
* Firmware compiles and flashes successfully.
* The heartbeat LED on PA6 blinks perfectly, indicating that the microcontroller is booting and running the main loop.
* The PlatformIO serial monitor on COM5 (9600 8-N-1) outputs continuous `\0` (NUL) bytes or garbage instead of readable text.

#### 🧠 Root Cause Analysis
1. **Physical Pin Mismatch (Solved):** In the previous configuration, the Picoprobe GP5 (RX) wire was connected to Pin 8. On the TSSOP20 package of the MM32G0001, PA12 (USART1_TX) is actually located on **Pin 17**. Moving the wire to Pin 17 established physical connection, but yielded only NUL bytes.
2. **Missing division by 16 in USART divisor:** The divisor calculation in `main.c` was set up as `8000000 / 9600 = 833.33`. However, the hardware UART baud rate register (`USART1_BRR`) expects `USARTDIV = Clock / (16 * BaudRate)`.
   - Programmed `USART1_BRR` divisor value: `833`
   - Due to the omission of the 16x oversampling factor, the actual baud rate generated by the hardware was: `8,000,000 / (16 * 833.3125) = 600.02 Baud`.
   - The serial monitor was reading at `9600 Baud`, leading to severe framing errors (start/stop bits misaligned) and receiving continuous `0x00` (NUL) bytes.

#### 🛠️ Resolution
1. **Relocate the Serial Probing Wire:** GP5 (RX) on the Picoprobe is connected to **Pin 17 (PA12)** on the MM32G0001 TSSOP20 package.
2. **Correct the Divisor Calculation in `main.c`:**
   - Re-calculated for HSI default of 8 MHz: `USARTDIV = 8,000,000 / (16 * 9600) = 52.0833`.
   - Integer part (MFD): `52` (`0x34`).
   - Fractional part (FFD): `0.0833 * 16 = 1.33 \approx 1` (`0x01`).
   - Configured `USART1_BRR = (52U << 4U) | 1U;`
3. **Outcome:** The generated baud rate aligns exactly with 9600 Baud, resulting in perfectly clear ASCII transmission on COM5.
